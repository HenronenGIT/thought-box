Production replacement

Thought Box Go Backend Pivot

Build a Go backend in apps/api-go as the future production backend. The Kotlin backend remains reference implementation until Go reaches parity.

Locked Decisions

Runtime

Go backend is production target. Kotlin backend stays temporarily as read-only reference.

Router

Use chi. Small, idiomatic, compatible with net/http.

Database

Use raw pgx. SQL remains explicit. No ORM.

Migrations

Use goose. Go app runs migrations on startup for parity with current behavior.

OpenAI

Use OpenAI Go SDK. Preserve model/prompt behavior unless a clear drop-in improvement appears.

Storage

Use AWS SDK for Go v2. Keep custom endpoint support for MinIO/local S3.

Worker

Keep in-process polling worker with Postgres row claiming and retry behavior.

Dev tools

Use root mise.toml for tool versions and local tasks. Makefile optional later.

API Contract

Frontend-facing API should remain compatible. Error messages may differ, but status codes and {"error": string} shape must match.

Route Behavior
GET /healthzReturn app health and environment.
GET /configReturn upload duration/size limits.
GET /meReturn seeded user id until real auth exists.
POST /thoughtsAccept multipart audio upload, validate, store in S3, insert pending thought.
GET /thoughtsList thoughts with cursor pagination and category/tag filters.
GET /thoughts/{id}Return one user-scoped thought.
GET /thoughts/{id}/audioStream audio bytes through backend.

Category Contract

Only these categories exist in DB, API, prompt schema, and frontend filters:

  • idea
  • observation
  • feeling
  • learning

Data And Config

DB URL

Go requires DATABASE_URL as a pgx-native Postgres URL containing username and password.

No JDBC fallback. Fail fast if missing or invalid.

Legacy

JDBC_DATABASE_URL may remain for Kotlin reference but Go ignores it.

Current data can be discarded. Migrations should define only the four new categories.

Pipeline

  1. Startup runs goose migrations.
  2. Startup recovery returns interrupted work to retryable states.
  3. Worker claims pending rows with FOR UPDATE SKIP LOCKED.
  4. Worker fetches audio from S3 and sends it to OpenAI transcription.
  5. Worker marks row enriching with transcript.
  6. Worker enriches transcript into category, tags, title, summary.
  7. Worker inserts enrichment and marks row done.
  8. Failures use same retry policy and terminal statuses as Kotlin.

Observability

  • Use log/slog structured logs.
  • Accept or generate X-Correlation-Id.
  • Include correlation id in logs and response headers.
  • Use custom CORS middleware.
  • Defer Sentry.

Local Dev Workflow

Run dependencies in Docker Compose. Run apps on host through mise.

# examples
mise run deps:up
mise run api-go:dev
mise run api-go:test
mise run web:dev

Deployment

Add a separate apps/api-go/Dockerfile. Deploy first to a separate Cloud Run service, for example thoughtbox-api-go. Switch frontend/API URL only after parity is verified.

Implementation Phases

  1. Scaffold apps/api-go, go.mod, package layout, Dockerfile, mise tasks.
  2. Implement config parsing, logging, CORS, health/config/me routes.
  3. Port migrations with goose and new category constraint.
  4. Implement domain models, validation, repository with raw pgx.
  5. Implement S3 blob store and multipart upload flow.
  6. Implement list/detail/audio routes.
  7. Implement OpenAI transcriber and enricher.
  8. Implement worker, retry policy, startup recovery.
  9. Add tests after implementation reaches full parity.
  10. Deploy to separate Cloud Run service and run parity QA.

Acceptance Criteria

  • Existing web app can point at Go API without frontend contract breakage.
  • Audio upload creates a pending thought and stores audio in S3/MinIO.
  • List/detail/audio endpoints work for seeded user.
  • Worker transcribes, enriches, retries, and reaches terminal states correctly.
  • Only idea, observation, feeling, learning are emitted or accepted as categories.
  • Go app fails fast when required env is missing, especially DATABASE_URL.
  • Cloud Run deploy uses separate service before production cutover.
  • Tests cover config, validation, repository, handlers, and worker transitions after implementation.
