Thought Box Kotlin Repository Guide

This document explains the Kotlin backend and the repository structure created for Thought Box: how the app boots, where each responsibility lives, how requests move through the code, and what files matter while learning Kotlin/JVM development.

Purpose

The repository is a small production-shaped Kotlin service plus a Next.js PWA. The backend receives dictated audio, stores it, transcribes it with Whisper, enriches the transcript with GPT, stores metadata in Postgres, and exposes JSON endpoints to the frontend.

Core learning axis: Kotlin + Gradle + Ktor + coroutines + raw SQL + JVM deployment. No ORM, no official OpenAI SDK, no queue, no integration tests in v1.

Runtime Path

The service starts in Application.kt. That file is the composition root: it creates concrete implementations and passes them into route/pipeline code.

1.Config.fromEnv() reads environment variables and validates required settings.
2.Ktor Netty starts on PORT.
3.Content negotiation, CORS, request logging, and Sentry are installed.
4.Flyway applies SQL migrations from src/main/resources/db/migration.
5.Hikari creates a Postgres connection pool.
6.Concrete modules are wired: repository, S3 store, Whisper transcriber, GPT enricher.
7.StartupRecovery unsticks interrupted rows.
8.ThoughtPipeline starts as a coroutine worker.
9.Ktor routes begin serving HTTP requests.

Kotlin Packages

entrypoint

Application.kt

Main function and app wiring. This is where interfaces become concrete classes. Good first file when asking: "what runs when the app starts?"

config

config/Config.kt

Converts raw env vars into typed Kotlin data classes. Elsewhere in the app should use Config, not System.getenv().

domain

domain/Models.kt

Core vocabulary: Status, Category, Thought, ThoughtEnrichment, and audio types. Also contains the status state machine.

http

http/Routes.kt, Dto.kt

Ktor handlers. Thin layer: parse request, call repository/storage, respond with DTOs. Business rules stay in validators, repository, and pipeline.

db

db/Database.kt, ThoughtRepository.kt

Flyway migration runner, Hikari connection pool, and raw SQL via kotliquery. This is the database boundary.

storage

storage/BlobStore.kt, S3BlobStore.kt

Interface plus AWS S3 implementation. MinIO works locally through the same code path by setting S3_ENDPOINT.

transcription

transcription/

Transcriber interface and OpenAiTranscriber. Calls Whisper with Ktor HttpClient, not the official SDK.

enrichment

enrichment/

Enricher interface and OpenAiEnricher. Calls chat completions with a JSON schema and returns category, tags, title, summary.

pipeline

pipeline/

Background coroutine worker. Moves thoughts through statuses, retries failures, and recovers interrupted work on startup.

user

user/UserResolver.kt

Single future auth swap point. v1 always returns the seeded UUID. Routes should not invent their own current-user logic.

validation

validation/Validators.kt

Small pure functions for MIME, duration, size, cursor, and limit parsing. Easy to test, safe to reuse.

observability

observability/Observability.kt

Request correlation IDs, structured request logs, and Sentry exception reporting. Avoid logging transcript text, prompts, audio bytes, or secrets.

Database

Flyway owns schema changes. Files run in version order:

MigrationCreatesWhy it matters
V1__users.sql users table + seeded UUID Prepares for future auth without redesigning the schema.
V2__thoughts.sql thoughts table Stores audio metadata, transcript, status, retry state, timestamps.
V3__thought_enrichments.sql thought_enrichments table Stores LLM output separately from the raw thought row.

SQL lives in ThoughtRepository.kt. Queries always receive userId where user-visible data is fetched. That keeps multi-user isolation ready for later.

Pipeline

The pipeline is deliberately simple: one in-process coroutine loop, no external queue.

pending
  -> transcribing
  -> enriching
  -> done

transcribing -> failed_transcription
enriching    -> failed_enrichment

RetryPolicy.kt decides whether to retry or fail: 1s, 4s, 16s, then terminal failure. StartupRecovery.kt resets interrupted transcription work so a restart does not strand a row forever.

Important invariant: audio is stored before the row enters the pipeline. Enrichment only exists after transcription succeeds.

Tests

Tests are unit-only in v1. They focus on pure behavior and small boundaries:

  • ConfigTest: required env vars and defaults.
  • UserResolverTest: seeded user id.
  • ValidatorsTest: MIME, duration, and size validation.
  • StatusTest: legal/illegal status transitions.
  • RetryPolicyTest: retry schedule.
  • ListParsingTest: cursor and limit parsing.
  • EnrichmentParsingTest: structured enrichment JSON parsing.

Not tested yet: real Postgres SQL, S3/MinIO, Ktor route handlers, full pipeline orchestration. Those need integration tests later.

Frontend Touchpoint

apps/web is the Next.js PWA. It is not Kotlin, but it is the client that proves the backend works. It fetches /healthz, reads /config, records audio with MediaRecorder, uploads to POST /thoughts, polls thought status, and plays audio through GET /thoughts/:id/audio.

The frontend talks to the backend through NEXT_PUBLIC_API_BASE_URL. In local dev that defaults to http://localhost:8080.

What to Learn From This Repo

  1. Kotlin data classes: used for config, DTOs, domain models.
  2. Enums and state machines: Status, Category, Status.next().
  3. Interfaces as boundaries: BlobStore, Transcriber, Enricher.
  4. Coroutines: background work in ThoughtPipeline.
  5. Ktor: routing, JSON serialization, CORS, status pages, Netty server.
  6. Raw SQL: kotliquery repository instead of ORM abstractions.
  7. JVM packaging: Shadow JAR, Docker image, JDK/JRE split.
  8. Operational shape: env config, migrations, structured logs, Sentry, CI.
Best reading order: Application.kt, Config.kt, Models.kt, Routes.kt, ThoughtRepository.kt, then ThoughtPipeline.kt.
